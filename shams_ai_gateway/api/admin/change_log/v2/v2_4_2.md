## Version 2.4.2

### Fixes
- **`update_document` now supports child tables (#168)** — Updating child-table rows on a draft document used to silently fail (the field was set to a raw list and either errored on save or left the doc inconsistent). The tool now accepts a list of row dicts under the table fieldname and picks **patch** or **replace** mode automatically: any row carrying a `name` puts the call in patch mode (matched rows are updated, unmatched rows are appended, rows with `_delete: true` are removed; rows not mentioned are left untouched), and no `name` on any row triggers replace mode (clear and re-fill). Restricted-field filtering now extends to child rows using the child doctype's own SENSITIVE/ADMIN_ONLY sets.
- **`update_document` rejects direct child-doctype calls** — Calling `update_document` with a child-table doctype (e.g. `Sales Order Item`) used to "succeed" silently while leaving derived fields stale: the row's `qty` was saved but the row's `amount` and the parent's `total`/`total_qty`/`grand_total` never recomputed, because saving a child row in isolation skips the parent's `validate()` pipeline. The tool now resolves the parent doc and table fieldname and returns a structured `child_doctype_direct_update` error with a ready-to-retry suggestion pointing at the parent.
- **`create_document` no longer reports false-positive missing fields (#165)** — `reqd` fields populated by a doctype controller's `set_missing_values()` (Quotation's `conversion_rate`, `price_list_currency`, `plc_conversion_rate`; many ERPNext Sales/Purchase doctypes) were being flagged as missing because the pre-flight check ran before `doc.insert()`. The pre-flight check is removed; Frappe's own `_validate_mandatory()` (which runs after `set_missing_values()`) is now the authority. `MandatoryError` is caught and translated into the same structured `missing_required_field` response shape with genuine missing-field names.
- **`create_document` no longer crashes with `UnboundLocalError` on Link validation** — A regression in the `MandatoryError` handler shadowed the module-level translation function `_`, breaking the later `_("Document Creation Error")` call inside `frappe.log_error` whenever execution reached the generic exception branch (e.g. an invalid Link reference). Fixed by not unpacking into `_` at all.
- **Audit log no longer redacts token-count metrics** — `output_data` on Assistant Audit Log was rendering `input_tokens` / `output_tokens` / `total_tokens` / `tokens_used` as `***REDACTED***` because the substring blocklist designed to catch `access_token` / `api_token` accidentally matched any key containing the word `token`. The blocklist is now regex-backed: it only matches `token` when preceded/followed by a non-letter, so credentials are still redacted but token-count metrics come through.
- **`chatgpt_fetch` no longer crashes on every call** — A signature change to `filter_sensitive_fields` (added a required `user_role` argument) was applied to the MCP-path callers but missed `chatgpt_fetch`, which raised `TypeError: filter_sensitive_fields() missing 1 required positional argument: 'user_role'` on every call. The tool now goes through the standard `validate_document_access` flow (consistent with `get_document`) and passes the resolved role through correctly. A misleading catch-all that masked the original `TypeError` as a generic "Error fetching document" was also removed.
- **`update_tool_role_access` accepts JSON-string roles from the admin UI** — Frappe's frontend serializes nested arrays as JSON strings over form encoding, but the strict `list` type hint caused pydantic to reject the payload before the function body's existing `json.loads` shim could run. Widening to `list | str` restores the admin-tools role-save flow (was failing with `417 EXPECTATION FAILED`).
- **HRMS Skill schema mismatch on `bench migrate`** — The v2.4.1 repair patch hard-coded reads/writes of a `skill_name` column that exists on some HRMS releases but not others. `bench migrate` was failing with `Unknown column 'skill_name' in 'SELECT'` on newer HRMS schemas. The patch now probes both `tabFAC Skill` and `tabSkill` for the column at runtime and includes it conditionally, so it's a no-op safe across HRMS schemas.

### Improvements
- **Conventional-commit linting and pip-audit available locally** — Added pre-commit hooks for commitlint (commit-msg stage) and pip-audit (manual stage), mirroring the CI checks. See `docs/development/PRE_COMMIT_SETUP.md` for setup. CI's commitlint job is also pinned to v19 to keep working on Node 18.
